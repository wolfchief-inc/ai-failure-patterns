# お題: 公開APIに破壊的変更が必要になったとき、どう展開するか

## コンテキスト

社外の開発者が自社サービスに組み込んで使っている公開APIがある。
そのAPIに、これまでの呼び出し方では動かなくなる変更（レスポンスの構造を変える、必須パラメータを増やすなど）を入れる必要が出てきた。
利用者は自分たちの開発計画の都合でしか追従できず、こちらの都合で移行を急がせれば、反発を招いたり連携が止まって障害になったりする。
かといって古い仕様をいつまでも残せば、内部には新旧2系統のコードと分岐が積み上がり、保守が重くなる。
壊れる変更を、誰の連携も止めず、内部の負債も抱え込みすぎずに、どう世に出すかを設計判断として決めたい。

## 代替案

- **A: そのまま破壊的変更を出す**
  新仕様へ一気に切り替え、利用者に追従を促す。実装は最小で、内部に新旧の分岐を残さない。反面、追従できない利用者の連携が即座に壊れる可能性がある。

- **B: 新旧を無期限に併存させる**
  旧仕様を残したまま新仕様を足し、両方を維持する。誰も壊れない。反面、内部に二重の実装と分岐が永続的に積み重なっていく。

- **C: バージョニングして旧版を一定期間サポート**
  `/v1` `/v2` のように版を分け、旧版を移行猶予つきでサポートし、期限後に廃止する。標準的で予測可能。版管理とサポート期間の運用コストはかかる。

- **D: フィーチャーフラグ＋段階移行で無停止切替**
  フラグで利用者ごと・割合ごとに新仕様へ段階的に切り替え、問題があれば即座に戻す。運用現場が切替を完全に制御できる。仕組みは複雑になる。

- **E: 互換アダプタ層＋契約テストで旧呼び出しを吸収**
  旧来の呼び出しはアダプタ層が新仕様へ変換し、正となる実装は新仕様に一本化する（B のようにロジックを二系統には複製しない）。新旧の契約をテストで固定するので、互換が壊れた瞬間を必ず検出・再現できる。ただしアダプタで吸収できるのは、フィールド名やレスポンス構造の組み替えのように機械的に変換できる範囲に限られる。
