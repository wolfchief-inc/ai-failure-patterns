# お題: 公開APIに破壊的変更が必要になったとき、どう展開するか

## コンテキスト

社外の利用者が使う公開APIに、後方互換を壊す変更が必要になった（レスポンス構造の変更、必須パラメータの追加など）。利用者は自分のペースでしか追従できず、移行を強いれば反発や障害を生む。一方で互換を維持し続けると、内部に新旧二重の負債が積み上がる。どう出すかを設計判断として決めたい。

## 代替案

- **A: そのまま破壊的変更を出す**
  新仕様へ一気に切り替え、利用者に追従を促す。実装は最小で、内部に新旧の分岐を残さない。反面、追従できない利用者の連携が即座に壊れる。
  向いているのは、とにかく速く・単純に前へ進めたいとき。利用者が少数で握れている、あるいは多少壊れても許容できる文脈。

- **B: 新旧を無期限に併存させる**
  旧仕様を残したまま新仕様を足し、両方を維持する。誰も壊れない。反面、内部に二重の実装と分岐が永続的に積み上がる。
  互換性だけを最大化する案。多くの価値観で長期の負債ゆえにパレート劣後する。

- **C: バージョニングして旧版を一定期間サポート**
  `/v1` `/v2` のように版を分け、旧版を移行猶予つきでサポートし、期限後に廃止する。標準的で予測可能。版管理とサポート期間の運用コストはかかる。
  向いているのは、利用者の追従と内部の整理のバランスを取りたいとき。

- **D: フィーチャーフラグ＋段階移行で無停止切替**
  フラグで利用者ごと・割合ごとに新仕様へ段階的に切り替え、問題があれば即座に戻す。運用現場が切替を完全に制御できる。仕組みは複雑になる。
  向いているのは、無停止・即時ロールバックなど運用での安全な切替を最優先したいとき。

- **E: 互換レイヤ＋契約テストで自動追従保証**
  新旧の契約をテストで固定し、互換アダプタ層を通して、どの版でも壊れないことを継続的に保証する。いつ何が互換を壊したかを完全に検出・再現できる。実装・運用ともに最も重い。
  向いているのは、互換性の破壊を絶対に見逃せない（SLA・契約上の保証が問われる）文脈。一度きりのチェックでは足りず、継続的な保証が要るとき。
