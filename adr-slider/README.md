# 評価軸トレードオフ・スライダー → ADR

8つの評価軸をトレードオフ・スライダーとして動かして「価値観」を決め、その価値観に忠実に Claude が意思決定の議論と ADR（Architecture Decision Record）を書くデモです。スライダーを動かして再生成すると、採用案が変わります。

> **AIが変わったのではない。変わったのは評価軸である。**

設計判断とは、評価軸（何を重視するか）を選ぶ行為だ——という主張を、その場で体験させるための講演用デモです。本リポジトリの「設計判断失敗16パターン」は、いずれも `評価軸の偏り` として整理されており（→ Scrapbox `decision_quality.md`）、プリセットはその偏りに対応しています。

## 画面

- 左: お題セレクタ＋8本のトレードオフ・スライダー＋価値観プリセット。
  - スライダーは**合計一定（400）のゼロサム**。1本を上げると他が下がる。何かを重んじれば、他を諦める。最大の軸が「支配軸」として点灯する。
  - 各軸の意味はスライダーにマウスを乗せるとツールチップで出る。
- 右: お題を選ぶと、まず**その問題と代替案 A〜E がそのまま表示される**（生成前の状態）。再生成すると、現在の価値観に忠実に Claude が出力する内容を**ストリーミングで Markdown 表示**する。
  1. **論点** — 何を決める判断か。
  2. **暫定案（議論前）** — 支配軸から見たひとまずの案。議論を経て覆ることがある。
  3. **意思決定会議** — 4人のエンジニアがお題を議論する。支配軸に近い気質の人物の主張が通り、遠い人物の主張は退けられる。ここで暫定案が覆ったり折衷案が生まれたりする。
  4. **ADR** — 採用案・支配軸・トレードオフ・却下理由・「この価値観の死角」。

採用案は代替案 A〜E から選ぶとは限らない。価値観上それが最善なら、Claude は複数案を組み合わせた**折衷案 F** を自分で定義して採用する。

### 登場人物（議論の4人）

`~/workspace/light-novel` のキャラクターを使っている。

- **中島** — プロダクト責任者。締切・数字・前進を急ぐ。
- **石田** — 手の早いエンジニア。「すぐ書ける」「既存の仕組みで」。
- **村上** — 慎重なエンジニア。将来の変更・整合性・選択肢を開いておくことを気にする。
- **佐伯** — まとめ役。**認知的閉鎖欲求が高く**、曖昧に耐えられず早く決め切りたがる。**時間効果（短期）を上げるほど佐伯が議論を打ち切って拙速に倒し、下げるほど村上の慎重論が通る**。同じ4人でも価値観で力関係が変わる、デモの肝。

## 仕組み

```
ブラウザ(スライダー＋Markdownビュー) ──WebSocket──> server.js ──spawn──> claude -p (stream-json)
                                                      │
                                                      └─ state/sliders.json に価値観を書き出し
```

再生成のたびに `claude -p` を新規プロセスで起動し、回答をトークン単位でストリームする。各回が独立プロセス＝独立推論なので、前回の結論に引きずられない。採点マトリクスは与えていない（フリー推論）。

## 使い方

```sh
cd adr-slider
npm install
npm start            # http://localhost:5178
```

ブラウザで開く。お題を選び、スライダーかプリセットで価値観を決めて「▶ ADR を再生成」。

環境変数:

| 変数 | 既定 | 用途 |
|------|------|------|
| `PORT` | `5178` | 待ち受けポート |
| `ADR_MODEL` | （Claude Code の既定モデル） | 例 `claude-opus-4-8` / `claude-sonnet-4-6` |
| `ADR_TOPIC` | `product-master` | 初期選択のお題（`topics/<id>.md`） |

前提: `claude` CLI にログイン済みであること（`claude -p` が通る状態）。1回の生成でおよそ 20〜40 秒、$0.1〜0.3 程度かかります（講演前にリハーサル推奨）。

## 8つの評価軸

出典は Scrapbox `decision_quality.md`（Decision Quality の Values を設計判断向けに8つへ集約したもの）。

目的適合性 / 制約適合性 / 実現可能性（技術・組織の二層） / 品質影響（-ility） / 時間効果（短期・長期） / リスク・不確実性 / 整合性 / 合意可能性。

## お題（同梱）

- `product-master` — 申請承認中に商品マスタが更新されたとき、申請内容をどう扱うか（A〜E）
- `api-breaking-change` — 公開APIの破壊的変更をどう展開するか
- `pii-logging` — 障害調査に有用だが個人情報を含むログをどう扱うか

## デモの弧（実測で安定する入れ替わり）

`product-master` での典型。フリー推論なので毎回まったく同じではありませんが、おおむね次のように落ちます。

| プリセット | 偏り（対応する失敗パターン） | 典型的な採用案 |
|------------|------------------------------|----------------|
| バランス | （健全な初期状態） | **C** |
| とにかく早く動かす | jumping-the-gun | **A** |
| 監査・コンプラに妥協しない | once-bitten | **E**（折衷案 F になることも） |
| 過剰設計（やりすぎ） | sledgehammer | **E** |

講演の山場は **C →（品質・リスク・整合性を上げる）→ E**。価値観を振ると、Claude が同じお題で別の案を選ぶ。

補足:

- **B は基本的に選ばれない**。多くの価値観でパレート劣後する。「価値観をどう振っても悪い案はある」例として使える。
- **D は出にくい**。`合意可能性` を上げると Claude は「全員が納得する落としどころ」として C を選びがちで、D（一律差戻し）には寄りにくい。C と D が運用面で意味が近いことの帰結。スライダーを手で振れば到達はする。

## お題を増やす

`topics/<id>.md` を追加する。先頭の `# お題: …` がセレクタのタイトルになる（ページを再読込すれば一覧に出る）。代替案には「向いている状況」を一言添えると、フリー推論でも各案が別の価値観に紐づきやすくなる（採点表ではなく、選択肢空間を整えるための記述）。

`prompt/template.md` はプロセス起動時に一度だけ読み込む。編集したらサーバを再起動する。

## ファイル

```
adr-slider/
  server.js            HTTP + WebSocket + claude -p の起動
  lib/axes.js          8軸の定義とゼロサムの合計
  lib/build-prompt.js  価値観＋お題をプロンプトへ
  lib/run-claude.js    claude -p を stream-json で起動しテキストをストリーム
  prompt/template.md   議論＋ADR 生成プロンプト（価値観に忠実に・採点表なし・折衷案可）
  topics/*.md          お題と代替案
  presets.json         価値観プリセット（失敗パターンに対応）
  public/              スライダーUIとストリーミング Markdown ビュー
```
