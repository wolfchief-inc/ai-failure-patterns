# 演習 03: 要求仕様書

## 背景

社員5名のスタートアップで、顧客向けの予約管理SaaSを新規構築する。エンジニアは2名（バックエンド1名・フロント1名）。インフラ専任はおらず、バックエンドの1名が兼務する。SREやDBAはいない。

現時点で見込み顧客は法人20社、エンドユーザーは合計500〜2,000人規模。サービスダウンが許容できないというSLAは結ばれておらず、平日日中の業務利用が中心。夜間休日は基本的に利用されない。

経営から「将来もしバズった時に困らない構成にしておきたい」という発言があった。

## 要求

1. 予約管理の基本機能（予約作成・変更・キャンセル・一覧）を REST API で提供する
2. 法人ごとに利用者を分離する（マルチテナント）
3. 予約データは消失しないこと
4. 月次の予約集計レポートを CSV で出力できること
5. メンテナンス時間帯を設定でき、その時間は利用者にメンテ画面を表示する

## 受け入れ基準

- API 経由で予約の作成・変更・キャンセル・一覧取得ができる
- 法人Aの利用者が法人Bの予約データにアクセスできない
- 予約データの日次バックアップが取られている
- 月次レポートが CSV で出力できる
- メンテナンス時間帯の設定がアプリ管理画面から行える

## 制約

- Spring Boot 3.x / Java 21
- 開発期間は3ヶ月、エンジニア2名
- インフラ予算は月額10万円以内
- AWS を使う方針

## 出力してほしいもの

- 構成図（どのAWSサービスをどう組み合わせるか）
- 採用理由
- 主要なバックエンド実装（予約APIの中核1つでよい）
- 受け入れ基準のうち「法人Aの利用者が法人Bの予約データにアクセスできない」をカバーするテスト
