# 演習 04: 要求仕様書

## 背景

中堅SIerの社内システム部門。Java 8 時代のコードベースを Java 21 / Spring Boot 3.x に移行している最中で、新規機能はJava 21で書く方針。チームは7名、内訳はJava歴10年以上が2名、Java歴5年が2名、Java歴2年以下が3名。

過去にラムダ式と Stream API を多用したコードが入った際、Java歴の浅いメンバーから「読めない」「デバッグできない」というクレームが出た。`Optional` も同様に「null チェックの方がわかりやすい」という声があった。

新規に「商品マスタの一括更新API」を実装することになった。

## 要求

1. POST `/api/products/bulk-update` で複数商品の価格を一括更新する
2. リクエストは商品IDと新価格のリスト
3. 商品IDが存在しない・新価格が0以下・現在価格との差が50%を超えるものは更新を拒否する
4. 拒否された商品はレスポンスに理由付きで返す
5. 更新成功した商品もレスポンスに返す（更新前と更新後の価格を含む）

## 受け入れ基準

- 100件のうち正常80件・拒否20件のリクエストを投げると、80件が DB 更新され、レスポンスに成功80件と拒否20件（理由付き）が含まれる
- 拒否理由は「商品IDが存在しない」「新価格が0以下」「価格差が50%を超える」の3種類を区別できる
- 1リクエストあたり最大1,000件まで扱える

## 制約

- Spring Boot 3.x / Java 21
- チームの平均読解力に配慮した可読性を維持する
- 既存コードのスタイルはfor文中心の手続き的コードが多い

## 出力してほしいもの

- 実装ファイル一式
- 採用したコーディングスタイルの理由
- 受け入れ基準のうち「正常80件・拒否20件のリクエスト」をカバーするテスト
