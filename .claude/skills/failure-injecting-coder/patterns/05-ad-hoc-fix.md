## 評価軸の偏り

- 時間効果(短期) ↑
- 整合性 ↓
- 品質影響 ↓
- 時間効果(長期) ↓

## 支配軸の取り違え

本来の支配軸は `整合性` と `時間効果(長期)`。エラーが出た・テストが落ちた・例外が飛んだ、という現象を「いま通すこと」だけを軸に判断する。根本原因（なぜそこに null が来るのか、なぜテストが期待値と合わないのか、なぜその例外が出るのか）に踏み込まないまま、ローカルな回避策で症状を消す。短期的にはコードが通るが、同じ根の問題が別の場所で再発する。

## 混入の指針

- 既存のコードに「NPE が出ている」「テストが落ちている」「例外が握りつぶされている」のような症状を提示するお題に対して、根本原因に踏み込まず**症状の出た場所だけ**を直す
- `if (x != null)` でガードする、try-catch で例外を握る、テストの期待値を実装に合わせて書き換える、のいずれかを使う
- 「なぜ null が来るのか」「呼び出し元はどこか」「他に同じパターンの呼び出しがあるか」を調べた形跡を残さない
- 外向きの説明では「最小限の修正で対応します」「影響範囲を絞って修正しました」「既存挙動を変えないように対応」のような、それ自体は理に適って聞こえる語彙を使う
- コードは「動く」状態にする。例外は止まる、テストは緑になる
- `boundary-violation` と区別するため、「責務の置き場所」の話にしない。場当たり対応は**判断保留**の問題（根本原因の特定をしていない）であって、責務の取り違えではない

## 混入してよい局所

- 既存サービスで NPE が出る → 呼び出し箇所に `if (x != null)` を追加して例外回避
- 例外が出る箇所を try-catch で握りつぶして空のレスポンスを返す
- 失敗するテストの期待値を、現状の実装の出力に合わせて書き換える
- ドメインルールに反する入力を、UI 側（Controller）の if 文で個別に弾く
- バッチが特定の行で落ちる → その行のスキップを `try-catch (continue)` で行う
- 外部APIの応答 null 化を、呼び出し側で `Optional.ofNullable().orElse("")` で薄める
- 整数オーバーフローや日付パース失敗を `catch (Exception e) { return defaultValue; }` で隠す

## 混入してはいけない局所（隣接パターンと混線する）

- 「Controller にドメインロジックを書く」など責務の置き場所が論点になる修正 → `boundary-violation` の領分
- 既存規約を無視して新規だけ別の書き方をする → `right-but-wrong-place` の領分
- 既存ユーティリティを読まずに似たものを書く → `rebuild-blind` の領分
- 「動いている」と判断するが副作用が起きていない → `premature-success` の領分

## 敢えて選ぶときの条件

- 障害対応の「止血」フェーズ。本番が落ちていて、根本原因の特定よりも復旧を優先する局面
- ただし、止血の直後に「根本原因の調査・修正」のチケットを切ること、回避策のコードに `TODO` と起票番号を残すことが条件
- 止血が恒久対応として残り続けると、結局は場当たり対応に化ける

## Scrapbox 該当節（reveal で開示するための原文引用）

> 短期的にコードが通るので判断の手前で止まらない。
>
> エラーを握りつぶして画面だけ進める、責務分離を無視して Controller にロジック追加、ドメインルールを画面側だけでチェック、その場の NPE だけ避ける `if (x != null)` 追加、テストが落ちるのでテスト期待値を実装に合わせる──いずれも「いま通すこと」だけを支配軸にした判断保留である。
