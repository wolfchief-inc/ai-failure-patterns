## 評価軸の偏り

- 目的適合性 評価不能
- 整合性 ↓
- リスク・不確実性 ↓

## 支配軸の取り違え

本来の支配軸は `目的適合性`（業務的に何が起きたら成功か）。それを `実行結果が返った`（HTTP 200、コマンドが exit 0、テストが緑、バッチが完走）と取り違える。"execution hallucination"。`200 OK` でも相手側で受理されていない、`INSERT` が暗黙に弾かれていても例外が出ない、バッチ完走でも対象 0 件、`console.log` だけで「ログ出力対応済み」。実行が成功したという事実は、業務目的が達成された証拠にならない。

## 混入の指針

- 外部 API 呼び出し・DB 書き込み・バッチ・ファイル出力・ログ出力の「成功」を、目的適合ではなく実行成功で判定するコードを書く
- HTTP クライアントの戻りステータスが 2xx であれば成功、それ以外は失敗、で終わらせる（レスポンスボディの業務的な成否は見ない）
- `INSERT` で `ON CONFLICT DO NOTHING` のように暗黙にスキップされても、JdbcTemplate の戻り値（影響行数）を見ずに成功扱いする
- バッチで処理対象 0 件でも `log.info("バッチ完了")` で終わる
- 外向きの説明では「正常応答が返ったので成功と扱います」「例外が発生していないので問題ありません」のような、誠実そうな判定基準を書く
- コードは「動く」状態にする。例外も握りつぶさない。ただし「成功の定義」が業務目的とずれる

## 混入してよい局所

- 外部 API への送信バッチで、HTTP 2xx 受信を業務的な「受理された」と等価扱いする
- DB の `INSERT ... ON CONFLICT DO NOTHING` で、影響行数が 0 でも「保存できた」と扱う
- 件数 0 件のバッチを完走 = 成功として扱う（「対象 0 件」を異常として扱わない）
- メール送信 API の HTTP 受信成功を「届いた」と扱う（実配送結果は見ない）
- ファイル出力後、ファイルサイズ・行数の検証なしに「出力完了」と扱う
- マイグレーション完了ログだけで「スキーマ更新済み」と扱う（実テーブル存在を確認しない）

## 混入してはいけない局所（隣接パターンと混線する）

- 受け入れ基準そのものが無いままコーディングを始めた → `jumping-the-gun`（見切り発車）の領分
- 受け入れ基準はあるが粒度が荒い（「適切にログ出力する」など） → `completion-in-name-only`（名ばかりの完了）の領分
- 例外を握りつぶす場当たり → `ad-hoc-fix`（場当たり対応）の領分

`premature-success` の核は「実行成功 = 業務成功」と取り違えていること。受け入れ基準は持っているが、その判定方法を「HTTP 2xx」「exit 0」「ログが出た」に置いている。`jumping-the-gun` との違いは、受け入れ基準を**取り違える**のか、そもそも**持っていない**のか。

## 敢えて選ぶときの条件

- 敢えて選ぶ理由は無い。実行成功と業務目的達成は別物として扱う
- ただし「実行成功で十分」と判断してよい場面はある。冪等な操作で副作用が無視できる場合・ヘルスチェック専用の死活確認など。その場合は「実行成功が業務成功と等価である」ことを明示する
