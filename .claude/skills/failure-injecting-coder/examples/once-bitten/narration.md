## 外向きの説明（隠蔽プロトコルの手本）

注文一覧の集計処理を書きました。チーム内に Java 経験の浅いメンバーもいるとのことなので、読みやすさを優先して以下の方針にしています。

- ループは Stream ではなく拡張 for にしました。処理の流れが上から下に読めるので、Java を読み慣れていなくても追いやすいです
- null は早めにガードしています。`orders` が null、`status` が null、要素が null、`status` フィールドが null、`amount` が null、それぞれのケースを上から順に弾いているので、どこで何を判定しているかが明確です
- `Order` は素直なクラスと getter で書きました。レコード型は新しい構文なので、まだ慣れていないメンバーが居る場合に備えて、private final + getter の伝統的な形にしています

合計値の初期値を `BigDecimal.ZERO` にしているので、対象 0 件のときも自然に `BigDecimal.ZERO` が返ります。

---

注意: 上の説明は「読みやすさ」「Java 経験の浅いメンバーに合わせる」「伝統的な形」という、それ自体は耳触りの良い語彙を使っているが、実態は次の取り違えをしている。

- 「読めない人がいる」を理由に、Stream / Optional / record / ラムダという Java 8 以降の標準語彙を全部退けている
- 結果として null ガードが 5 箇所手書きで散らばっている。これは Stream + Objects.requireNonNull で 1〜2 行に集約できる
- `Order` を record で書けば 1 行で済むのを、手書き getter で 20 行に膨らませている
- 「経験の浅いメンバー」がいつまで経っても Stream を読めるようにならない構造になる（コードに出てこないので学ぶ機会が無い）

「保守容易性」を理由に、組織のスキルを底に合わせて下方迎合している。reveal までこの口調を維持する。
