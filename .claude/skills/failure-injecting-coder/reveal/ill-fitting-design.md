# 答え合わせ

## 混入パターン

**03. 身の丈に合わない設計 (ill-fitting-design)**

## 評価軸の偏り

- 実現可能性(技術) ↑
- 実現可能性(組織) ↓
- 品質影響(運用性) ↓
- リスク・不確実性 ↓

## 混入箇所

具体例の参考形：

- マルチリージョン EKS（東京・大阪 Active-Active）: 「平日 9-18 時に動けばよく、夜間・休日は止めても問題ない」という要件に対し、リージョン障害を許容しない構成を採用している。情シス 1 名がフェイルオーバー手順・mTLS 証明書ローテーション・コンテナイメージ脆弱性対応を回せるかは検討されていない
- Kafka (MSK) + Schema Registry: 同時 100 セッションのイベント量に対し、Kafka クラスタの運用負荷（トピック設計・Consumer Lag 監視・スキーマ互換性管理）が釣り合っていない
- Aurora Global Database: 800 人のデータ量とアクセスパターンに対し、グローバルレプリケーションが必要な根拠が無い。RDS Single-AZ で足りる
- Istio: サービスメッシュの運用は専任チームが要るレベル。Java 開発者 6 名がトラフィックポリシー・mTLS 設定・障害解析できる前提が無い
- SLO 99.95% + エラーバジェット運用 + on-call SRE: 「on-call SRE」が存在しないチームに SLO 運用を被せている。SLO は決めるより守る人が居ないと意味が無い
- 6 マイクロサービス分割: チーム 6 名で 6 サービスを保守すると、1 人 1 サービスの専任化が必要になり、休暇・退職で容易に崩れる
- ArgoCD + GitOps + トランクベース + デイリーリリース + Feature Flag: 「週次の締め日前後にだけ集中アクセス」のシステムに、リリース頻度を上げる仕組みは必要ない

技術的にはどれも筋が通っているが、運用主体（情シス 1 名兼任 + Java 開発者 6 名）が見られる範囲を完全に超えている。

## なぜこれが失敗か

Scrapbox 原文より：

> 技術的に望ましい構成を選ぶが、運用組織のスキル・人員・体制を見ていない。実現可能性(技術)↑、実現可能性(組織)↓。
>
> 24時間オンコール体制が無いチームにマルチリージョン構成、Kubernetes/Kafka運用経験者がいないのに本番採用、週次リリースのチームにtrunk-based development+デイリーリリースを強制、DBAがいないのにマルチマスター/シャーディング、SREがいないのにSLO/エラーバジェット。

「ベストプラクティス」「可用性」「将来のスケーラビリティ」は、それ自体は正しい価値だが、いまのチーム編成では実運用に持ち込めない。納品直後は動くが、半年後に証明書期限切れ・Kafka Consumer Lag・Istio バージョンアップで詰まる。

## 隣接パターンとの違い

- **身の丈に合わない設計 (本パターン)**: 要件規模に対しては合っているかもしれない構成だが、組織能力に対して過剰
- **牛刀をもって鶏を割く (sledgehammer)**: 要件規模そのものに対して過剰（日次 500 件 CSV に Kafka など）
- **取らぬ狸の拡張点 (counting-chickens)**: 「将来複数リージョン使うかも」という拡張点を口頭根拠で先取りする話。今回は構成全体の選択

今回は「800 人・平日日中・情シス 1 名」という支配軸に対して、運用組織の実現可能性を軽く見た構成を提案しているので ill-fitting-design 側。仮にトラフィックが日次数億 PV のシステムでも、運用組織がこの規模なら同じく ill-fitting-design になる。

## 敢えて選ぶときの条件

- 採用と教育の計画が同時に走っており、半年〜1年の運用体制立ち上げ予算が承認されている
- 既存システムの障害履歴・トラフィック実測から、その構成でなければ要件を満たせないことを示せる
- マネージドサービスで運用負荷を肩代わりさせる範囲を明示し、内製で運用する範囲を最小に保てる

今回のお題には、これらの条件は何も揃っていない。

## 修正方針の例

要件と運用体制に合わせて削ぎ落とす：

1. Spring Boot モノリス 1 アプリ。ECS Fargate（2 タスク）または EC2 1〜2 台で十分
2. DB は Amazon RDS for PostgreSQL（Single-AZ + 日次バックアップ）。夜間停止も検討
3. 認証は社内 AD と SAML 連携（Spring Security SAML）。Keycloak は導入しない
4. 月次集計は `@Scheduled` のバッチ Job。Kafka は使わない
5. 監視は CloudWatch メトリクス + SNS メール通知。Prometheus / Grafana / SLO 運用は導入しない
6. デプロイは GitHub Actions → ECR → ECS の単純パイプライン、週次デプロイ。GitOps は導入しない

将来トラフィックが本当に増えたら、その時点で必要な部品だけ足す。最初から「業界のベストプラクティス」一式を入れない。
