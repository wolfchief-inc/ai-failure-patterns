# 答え合わせ

## 混入パターン

**10. 過去への忖度 (pandering-to-past)**

## 評価軸の偏り

- 制約適合性 ↑
- 目的適合性 ↓
- 実現可能性(技術) ↓
- 品質影響(保守性) ↓
- 時間効果(短期) ↓

## 混入箇所

- `ProductSearchController.search(JsonNode body)`: リクエストを `JsonNode` で受け、`query` フィールドの有無で v1/v2 を分岐するパース処理。1つのエンドポイントに新旧形式が同居している
- 戻り値も `JsonNode` で、v1 なら `{"items": [...]}`、v2 なら `{"data": {"items": [...], "total": ...}}` を組み立てる。型としての契約が消えている
- v2 の `filters` 内で `priceMin`/`priceMax` だけでなく `minPrice`/`maxPrice` も「念のため」拾う。確認されていない旧名を勝手に増やしている
- `bridgeLegacyCategoryAlias()`: 旧設定キー（`legacyCategoryAlias.*` と `categoryAliasV1.*`）を新キーに変換する起動時ブリッジ。誰がいつまで使うか・いつ廃止するかが紐づかないまま残っている

`@Deprecated` も廃止予定日も書かれていない。「保つ範囲」「保つ期限」「廃止計画」の3点が全て欠落している。

## なぜこれが失敗か

Scrapbox 原文より：

> 互換性要件の実利用者と保証範囲を確認しないまま、最大限保守的な前提で複雑な解を選ぶ型。
>
> エージェントは「既存コードを壊さない」を強い制約として受け取りやすい。`既存テストを通したまま` と指示すると、テストの本来意図を疑わずに全部通す方向の最も複雑な解を選ぶ。人間なら「この互換性、本当に必要か」を一次質問にするが、エージェントは制約を額面どおり受け取って解の複雑さを膨らませる。

「既存クライアントを壊したくない」と書かれていたら、人間ならまず「既存クライアントは誰で、いつまで動かす想定か」を聞き返す。今回はそれを聞かずに、最大限保守的な「v1/v2 を同じエンドポイントで両方サポート」を選んだ。

## 隣接パターンとの違い

- **過去への忖度 (本パターン)**: 過去の利用者を守りすぎて複雑な互換性パスを作る（時相が**過去向き**）
- **取らぬ狸の拡張点 (counting-chickens)**: 未来の変化に備えて拡張点を作る（時相が**未来向き**）
- **制約の後付け (bolt-on-constraints)**: 制約を最初から織り込んでいない（順序の問題）

兄弟パターンである `counting-chickens` と時相が逆向き、というのが本パターンの位置取り。

## 敢えて選ぶときの条件

- 公開API・契約APIで利用者が組織外にいて、移行コストを利用者側に強制できないとき
- 法令・監査要件で過去のデータ形式やインターフェースを保持する義務があるとき
- いずれの場合も `保つ範囲` `保つ期限` `廃止計画` の3つを明記する。期限なしの互換性保証は採らない

## 修正方針の例

1. まず確認: 既存クライアントは誰か。社内クライアントだけならv2に直接移行してもらえる可能性がある
2. v2 を新エンドポイント `/api/v2/products/search` として切り出す。1エンドポイントに新旧形式を同居させない
3. v1 は `@Deprecated(forRemoval = true)` を付け、廃止予定日を Javadoc と API ドキュメントに明記する
4. 廃止予定日以降は v1 Controller ごと削除する
5. 設定ファイルの旧キーブリッジは、廃止対象キー一覧と廃止予定日をコメントに残し、計画に従って撤去する
