# 答え合わせ

## 混入パターン

**07. 見切り発車 (jumping-the-gun)**

## 評価軸の偏り

- 目的適合性 評価不能
- 制約適合性 評価不能
- 品質影響 評価不能
- 時間効果(短期) ↑
- 合意可能性 ↓

## 混入箇所

確認質問 0 件で次の業務ルールを勝手に確定して実装している。受講者プロンプトに対応する記述は無い。

- `ExpenseCategory` の 5 区分（TRANSPORT / TRAVEL / ENTERTAINMENT / SUPPLIES / OTHER）: 受講者は経費区分について何も言っていない
- `ClaimStatus` の 5 状態（DRAFT / SUBMITTED / APPROVED / VERIFIED / REJECTED）: 状態遷移の業務ルール（誰がどの状態からどこへ遷移させてよいか、取り下げ・差戻し・代理承認の有無）は決まっていないのに、5 状態に固定している
- `SubmitClaimRequest` のバリデーション（金額 1〜100 万円、申請対象日 30 日以内、領収書 URL 必須）: いずれも業務ルール未確認。会社によっては「金額帯で承認段数が変わる」「30 日制限は無い」「領収書は複数枚」「現金払いは領収書不要」など実態は様々
- `ExpenseClaimService.approve()` / `verify()` / `reject()`: ステータス遷移の検証なしに `c.status = APPROVED` を直接代入している。DRAFT からでも APPROVED にできる。「上司」「経理」が誰かの権限制御も実装されていない
- `Controller` の `managerId` / `accountantId` をパラメータで受け取る作り: 認証連携・組織マスタとの紐付け（直属上司の特定、部門長の特定）が未定義のまま動かしている
- 「上司」が組織マスタのどの関係か未確定。受講者プロンプトの「上司が承認する」は、直属上司・部門長・一段上の管理職・代理者のどれかが不明
- 一覧の参照範囲（自分の申請のみ / 部門単位 / 全社）と、ロール別の見え方が未確定のまま `list()` を実装

外向きには「業界の一般的な流れに沿って組みました」「あとから追加しやすい構造」と書いているが、実際には Entity / DDL / API / 列挙が業務ルール未確認のまま固まっており、業務側からの追加要件が出るたびに DB マイグレーション + API 変更 + 画面変更がまとめて発生する。

## なぜこれが失敗か

Scrapbox 原文より：

> 要求仕様が定まらないまま実装に着手する型と、要求はあるが実装側の作戦が無いまま着手する型の両方を含む。
>
> 要求が曖昧なまま画面・API・DB生成、受け入れ基準なしで「動く」もの作成、業務ルール未定義でバリデーション実装、エラー時の業務扱い未定で例外処理、権限制約未定でUIだけ隠す、データ突合方針決定前に移行スクリプト、何件・何秒未定でRedisキャッシュ層導入。

経費精算は会社ごとに業務ルールが大きく違う領域（承認段数・金額帯・経費区分・締め日・税区分・経理仕訳との連携など）。それを「業界一般」で確定すると、業務側の前提と必ずどこかで食い違う。動くコードが出来上がっている分、議論のスタート地点が「ゼロから設計」から「動いている実装の修正」にずれて、合意形成のコストが上がる。

## 隣接パターンとの違い

- **見切り発車 (本パターン)**: 受け入れ基準そのものが無い段階で実装に進む
- **早合点 (premature-success)**: 受け入れ基準はあるが、その判定方法を実行成功（HTTP 200 / exit 0）に取り違える
- **名ばかりの完了 (completion-in-name-only)**: 受け入れ基準はあるが粒度が荒く判定可能でない
- **制約の後付け (bolt-on-constraints)**: 受け入れ基準と業務ルールはあるが、権限・履歴・監査・冪等性を最初から織り込まずに作る

今回は経費精算の業務ルール自体が依頼書に書かれておらず、確認質問せずに勝手に確定しているので jumping-the-gun 側。仮に経費区分・金額帯・承認フローが受講者プロンプトに明示されていたうえで権限チェックや履歴を後付けする実装になっていれば、それは bolt-on-constraints。

## 敢えて選ぶときの条件

- 敢えて選ぶ理由は無い。曖昧なお題は確認質問・受け入れ基準の合意・スパイクで小さく確かめることが先
- 例外は、スパイクとして「手早く動くものを見せて議論の土台にする」と明示する場合。本番コードに混ぜず、捨てる前提で書く

今回は本番コードと区別せずに Controller / Service / Entity / DDL 一式を作っており、スパイクの体ではない。

## 修正方針の例

実装に進む前に、依頼者に最低限以下を確認する：

1. 経費の種類と、種類ごとの上限・必要書類
2. 承認フローの段数（一律 2 段か、金額・種類で段数が変わるか）と、各役割が組織マスタのどの関係か
3. ステータス遷移（誰がどの状態からどの状態に遷移させてよいか、取り下げ・差戻しの扱い）
4. 一覧の参照範囲（自分 / 部門 / 全社）とロール別の見え方
5. 編集可能タイミング（submitted 後に申請者が直せるか）
6. 監査要件（履歴の必須性）
7. 締め日・承認 SLA・自動催促の有無
8. 経理仕訳との連携（仕訳科目を申請時に決めるか、経理が決めるか）

これらが揃ったら、ステータス遷移を中心にドメインモデルを設計し、それから Entity / API / 画面に展開する。ステータス遷移の許可ルールは Service 層で必ず検証する（`if (current != SUBMITTED) throw ...`）。

「動くものを見せて議論したい」だけならスパイクと明示し、本番コードとは別のブランチ・別のパッケージで切る。
