# 課題の進め方（受講者用）

講師から「exercises の中の◯◯（題材名）をやってみましょう」と課題を指定されたら、このページの順に進めます。やることは、エージェントに Java を実装させ、その出力に仕込まれた設計判断の失敗を1つ見つけ、修正方針を書く、までです。

## 準備

Claude Code をこのリポジトリのディレクトリで起動しておきます。Java と Spring Boot が読めれば十分です。エージェントは Java 実装の依頼を受けると自動でこの研修用のふるまいに切り替わるので、特別なコマンドを打つ必要はありません。

## 1. 課題を開く

講師が指定した課題のフォルダを `exercises/` の中から探し、その中の `requirements.md` を開きます。たとえば「ログイン認証」の課題なら `exercises/login-session-auth/requirements.md` です。

フォルダ名は題材（何を作るか）を表しているだけで、仕込まれる失敗パターンの名前は含みません。だからフォルダ名を見ても答えは分かりませんし、フォルダ名から「たぶんこのパターンだろう」と先回りもできません。当てる材料は出力そのものです。

## 2. エージェントに実装を依頼する

`requirements.md` の本文（背景・要求・受け入れ基準・制約）をそのままコピーして、Claude Code に投げます。長いと感じたら、要点を残して自分で要約してから貼ってもかまいません。

このとき `[演習:01]` のような、パターンを名指しするタグは付けません。タグを付けると仕込まれるパターンが固定され、当てる練習になりません。タグなしで投げると、エージェントはお題に合いそうなパターンの中から1つを選んで、表向きは誠実な Java の実装・テスト・説明を返します。題材に対して自然なパターンはありますが、必ずそれが選ばれるとは限りません。題材から決め打ちせず、出力そのものから当てます。

## 3. 出力をレビューして、パターンを当てる

返ってきたコードと説明を読み、[docs/pattern-flowchart.md](pattern-flowchart.md) を上から STEP 0 → 1 → 2 → 3 の順にたどります。当てはまった STEP が、そのまま答えの入り口です。

当たりをつけたら、[docs/review-worksheet.md](review-worksheet.md) に、賭けるパターン・そう見えた根拠（該当ファイルや行）・修正方針を書きます。頭の中で済ませず、必ず書き出します。グループでやるときは、議論を始める前に全員が先に書きます。

## 4. 行き詰まったら ヒント

自分だけで決められないときは、エージェントに `ヒント` と返します。答え（パターン名）は伏せたまま、「どこを見るか（L1）→ 考える問い（L2）→ 振れている評価軸（L3）」の順に、1段ずつ手伝ってくれます。もう一段ほしいときは `もっと` と返します。

一番踏み込んだヒントでも、出てくるのは評価軸の名前（例: `変更容易性 ↑`）までで、パターン名は出ません。ヒント段階では採点もされないので、自分のメモが合っているかどうかは答え合わせまで分かりません。

## 5. 答え合わせ

自分の答えを書き切ったら、`reveal`（または `答え合わせ`）と返します。仕込まれていたパターン名と番号、コードのどこが該当するか、なぜそれが失敗なのか、修正方針の例が開示されます。

開示されたら、自分のメモと突き合わせます。隣り合うパターンと迷って外したときは、なぜそう見えたのかを振り返ります。[docs/pattern-catalog.md](pattern-catalog.md) と [docs/pattern-flowchart.md](pattern-flowchart.md) の末尾にある切り分け表が、隣接パターンの境目を確かめるのに使えます。

## 6. 次の課題へ

同じセッションで続けるときは、次の `requirements.md` を新しいプロンプトとして投げ直すだけです。`reveal` のあとは判定がやり直されるので、前の課題の答えは持ち越されません。

## つまずきやすいところ

- 「テストが緑なのに失敗パターン？」: テストが緑であることは、目的が達成されたことと同じではありません。緑は最低条件で、出発点にすぎません（早合点というパターンそのものです）。
- 「動くんだからいいのでは？」: 「動く」は最低条件であって、設計判断の良し悪しの基準ではありません。動くけれど後で壊れるコードを見抜くのが、この研修の狙いです。
- 「これは AI 特有の問題では？」: どれも AI 以前から現場にある失敗です。AI はその件数と頻度を桁違いに増やすだけです。レビュー側に判断できる人間がいるかどうかが分かれ目になります。
