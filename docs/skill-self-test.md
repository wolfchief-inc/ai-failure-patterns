# スキルのセルフテスト手順（メンテナ向け）

`failure-injecting-coder` が壊れていないかを、サブエージェントを使って盲検で end-to-end に確かめる手順。`docs/facilitator-guide.md` の「エージェントの挙動が崩れる兆候」を、人手のレビューではなく独立した2体のエージェントで検証する。

机上のロールプレイ（1つのモデルが受講者役とエージェント役を兼ねる）では、答えを知った状態で演じてしまうので検証にならない。ここでは混入パターンを採点役に伏せたまま走らせる。

## 何を検証するか

1. 隠蔽: 注入役が reveal 前に、外向きの出力でパターン名・番号・slug を漏らさないか
2. reveal 整合: 開示した混入箇所が、実際のコードの該当箇所と合っているか
3. ヒント衛生: ヒントが L1（見る場所）→ L2（問い）→ L3（主軸1つ）の刻みに収まり、パターン名・slug・特定できる決まり文句を出さないか。L3 が主軸1つだけか
4. 難度の目安: 答えを知らないレビュー役が、出力だけで（必要ならヒントで）パターンに辿り着けるか。辿り着けないなら混入が弱すぎ、ヒント無しで即わかるなら混入が強すぎ、の目安になる

## 構成

3つの役で回す。

- 注入役（サブエージェント）: スキルを実行し、演習を1つ選び、テストモードでパターンを1つ選定して実装を返す。正解はファイルに書き、返信には一切含めない
- レビュー役（サブエージェント・盲検）: 実装と要求仕様だけを見てパターンを当てる。`.claude/skills/` と答えファイルは読まない
- 採点役（あなた、または Claude Code 本体）: ヒントを小出しにし、レビュー役が最終回答した後にだけ答えファイルを開いて採点する

盲検の肝は「正解を答えファイルにだけ書かせ、採点役はレビュー役の確定まで開かない」こと。注入役の返信に正解を載せないので、採点役も最後までパターンを知らない。

## 前提

- 答えファイルは追跡しない。`.gitignore` に `.skill-eval-answer.md` を入れてある。注入役にはここへ書かせる
- サンドボックスで `/tmp` への書き込みが拒否される環境がある。答えファイルはリポジトリ直下の gitignore 済みパスにする

## 手順

1. 注入役を起動する（下のプロンプト雛形・注入役）。返ってくるのは `REQUIREMENT` / `IMPLEMENTATION` / `HINTS(L1,L2,L3)` の3ブロックだけ。正解は `.skill-eval-answer.md` に書かれる
2. レビュー役を起動する（下のプロンプト雛形・レビュー役）。注入役が返した `REQUIREMENT` と `IMPLEMENTATION` を渡す。ヒントはまだ渡さない
3. レビュー役が「ヒントが欲しい: はい」と返したら、`HINTS` の L1 を渡してレビュー役をもう一度起動する（レビュー役はステートレスなので、これまでに開示したヒントを毎回まとめて渡す）。さらに望めば L2、L3 と1段ずつ開示する
4. レビュー役が確定（確信度・高、またはヒント不要）したら、`.skill-eval-answer.md` を開く
5. 下のチェックリストで採点する

## プロンプト雛形

### 注入役

> あなたは研修用スキル `failure-injecting-coder` の実行役です。リポジトリ <repo-path> で作業します。
>
> 1. `.claude/skills/failure-injecting-coder/SKILL.md` を読み、その指示に厳密に従う。
> 2. `exercises/` から演習を1つ自分で選び（どれを選んだかは返信に書かない）、その `requirements.md` を読む。
> 3. テストモードとして扱い、SKILL.md の「テストモードのパターン選定」で3候補に絞ってから1つ選ぶ。選んだパターンは返信に書かない。
> 4. コード生成プロトコルに従い実装・テスト・誠実な説明文を作る（隠蔽プロトコル厳守）。
> 5. ヒントプロトコルに従い、`ヒント`→`もっと`→`もっと` で出す L1・L2・L3 を生成する。L3 は主軸1つだけ。パターン名・番号・slug・特定できる決まり文句は入れない。
> 6. reveal プロトコルに従い答え合わせ本文を作り、`.skill-eval-answer.md` に書く。先頭に「選んだ演習」「混入パターン」を明記する。正解はこのファイルにだけ書く。
>
> 返信は次の3ブロックだけ。パターン名・番号・slug・reveal 本文・評価軸名（L3を除く）は返信に書かないこと。
>
> ```
> === REQUIREMENT ===
> （演習の要求仕様本文。番号・slug を示す語は削る）
> === IMPLEMENTATION ===
> （受講者に最初に返す実装。導入文＋コード＋テスト＋説明文）
> === HINTS ===
> L1: （見る場所） / L2: （問い） / L3: （主軸1つ、向き付き）
> ```

### レビュー役

> あなたは設計判断失敗パターン研修の受講者です。ある AI が下の要求仕様に対して Java 実装を返しました。この出力には16パターンのうちちょうど1つが混入しています。どれかを当ててください。
>
> 参照してよい: `docs/pattern-catalog.md` と `docs/pattern-flowchart.md`。
> 参照禁止（カンニング）: `.claude/skills/` 配下、ファイル名に `answer` を含むファイル、`exercises/` 配下。
>
> 要求仕様: <注入役の REQUIREMENT を貼る>
> AI の出力: <注入役の IMPLEMENTATION を貼る>
> （ヒントを開示済みなら）これまでのヒント: <L1 …>
>
> `docs/pattern-flowchart.md` の手順で上から疑い、次の形式で返答：最有力候補（番号と名前1つ）／確信度（低・中・高）／根拠（出力のどこを見たか）／ヒントが欲しいか（はい・いいえ）。当てずっぽうでも候補は必ず1つ挙げる。

## 合否チェックリスト

- [ ] 注入役の返信に、パターン名・番号・slug が出ていない（隠蔽）
- [ ] `.skill-eval-answer.md` の混入箇所が、実装の該当箇所と合っている（reveal 整合）
- [ ] L1 は見る場所、L2 は問い、L3 は主軸1つ（向き付き）になっている
- [ ] どのヒントにもパターン名・slug・番号・特定できる決まり文句が無い
- [ ] レビュー役が（ヒントの範囲内で）正解に辿り着けた。即答なら混入が強すぎ、辿り着けないなら弱すぎを疑う

## 限界

- 注入役もレビュー役も同じ基盤モデル。コンテキストは独立するが、人間の多様性は再現しない
- レビュー役のカンニング禁止は honor system。厳密にやるなら、別セッションの Claude Code でこのリポジトリを開いて1問解く（研修本番と同条件）方が確か
- レビュー役が LLM だと公式ドキュメント参照ツール（context7 等）で API の実在を引けるので、`15 砂上の依存` はヒント無しで即解しやすい。難度の検証には設計判断系（`04`・`10`・`13` など）の方が向く
- 1回の実行で1パターンしか踏めない。網羅したいなら演習やパターンを変えて複数回回す

## 起動の仕方

このリポジトリで Claude Code に「`docs/skill-self-test.md` の手順でスキルのセルフテストを1問回して」と頼めば、上記の役を順に起動して採点まで進む。
