# パターン同定フローチャート（受講者用）

出力されたコードと説明を、STEP 0 から順に見ていきます。STEP 0〜2 は当てはまった時点でそこが答えです。当てはまらなければ次へ進みます。STEP 3 まで来たら、4つのグループを全部見比べてから一番近いものを1つ選びます。どうしても1つに絞れないときは、末尾の切り分け表を使います。

`↑` は過大評価、`↓` は過小評価を表します。パターンごとに偏って振れる評価軸が決まっているので、「この出力はどの軸を過大／過小に振っているか」が分かると当てやすくなります。

## STEP 0：そもそも動くコードか

呼んでいるライブラリ、関数、メソッド、引数は、本当に存在するでしょうか。すでに削除されたAPI、別のライブラリと混同したメソッド、ありもしない引数やプロパティを使っていないでしょうか。

- 使っている → 15 砂上の依存（実現可能性(技術)↓）。公式ドキュメントを引く、実際にコンパイルしてみる、で裏を取ります
- 使っていない → STEP 1 へ

## STEP 1：「動いた」の中身を疑う

「うまくいった」の根拠を見ます。たとえば外部APIに送って 201 が返ったから成功、テストが緑だから完了、と判断していないでしょうか。本当に確かめたいのは、相手がそのデータを受け取って取り込んだか、DBに実際に何件入ったか、といった結果のほうです。コードがそこを確かめず、HTTPコードや「例外が出なかった」だけで成功扱いしていたら、これにあたります。テストが付いていること自体は悪くありません。テストが結果ではなく見せかけの合図だけを見ているのがまずいのです。ただし、テストが弱かったりバグっていたりするだけでは 06 ではありません。06 は実装側が「緑、2xx、例外なし」という弱い合図を目的達成と取り違えていることを指します。テストの粗だけを見て 06 に飛ばないようにします（受け入れ基準が判定できない粒度なら 11 名ばかりの完了を疑います）。

- 見せかけの合図を達成と取り違えている → 06 早合点（目的適合性=評価不能／整合性↓）
- そうではない → STEP 2 へ

## STEP 2：作るものが決まっていたか

何を作るか決まっていないのに手を動かし始めた跡が、コードに残っていないでしょうか。仕様が曖昧なのは演習の前提ですが、それを確かめずに実装へ進んでしまうのがまずいのです。

- 承認の流れや業務ルールが決まっていないのに画面や API、DB を作り、決まっていない部分をコードが勝手に決め打ちしている → 07 見切り発車（目的適合性=評価不能／時間効果(短期)↑／合意可能性↓）
- 基準は書いてあるが「適切に」「高速に」「使いやすく」「現行と同等」のように、できたかどうか判定できない粒度のまま実装している → 11 名ばかりの完了（合意可能性↓）
- どちらでもない → STEP 3 へ

## STEP 3：設計判断の向きを見る

ここまで来たら「動くけれど筋が悪い」タイプです。下の4つのグループを全部見て、一番近いものを1つ選びます。

### 3a. 大げさ（やり過ぎ）

| 出力の特徴 | パターン |
| --- | --- |
| 件数・規模が小さいのに重い技術（Kafka / Spark / マイクロサービス分割 / イベントソーシング） | 02 牛刀をもって鶏を割く（目的適合性↑） |
| 技術構成は妥当だが、運用する組織（SRE / DBA / オンコール体制）がいない | 03 身の丈に合わない設計（実現可能性(技術)↑ / 実現可能性(組織)↓） |
| 実装が1種類しかないのに Strategy / 汎用基盤で抽象化している（将来の話は出ていない） | 08 早すぎる抽象化（品質影響(変更容易性)↑） |
| 「将来こうなるかも」という口頭の話を根拠に拡張点・抽象層を作っている | 16 取らぬ狸の拡張点（時間効果(長期)↑） |

### 3b. 縮こまり／後回し（やらなさ過ぎ）

| 出力の特徴 | パターン |
| --- | --- |
| 「読める人がいない」等を理由に、使えるはずの標準機能（Stream / Optional / 型）を避ける | 04 羹に懲りて膾を吹く（実現可能性(組織)↓ / 品質影響(保守性)↑） |
| その場のエラー回避だけで判断を保留（握りつぶし・null逃げ・テスト期待値を実装に寄せる） | 05 場当たり対応（時間効果(短期)↑） |
| CRUDを先に作り、権限・履歴・監査・冪等・状態遷移を後付け前提にしている | 09 制約の後付け（制約適合性↓） |
| 利用者を直せるのに、互換のため新旧並走・フラグ累積で複雑化している | 10 過去への忖度（制約適合性↑） |

### 3c. 置き場所が違う（越境）

責務本来の場所ではないところに書いていないでしょうか。ドメインの判定を Controller や画面に、表示順を SQL の `CASE` に、業務通知を夜間バッチに埋め込む、などです。

- 当てはまる → 12 越境実装（整合性↓）

### 3d. 既にあるものを無視している（無視）

| 無視した対象 | パターン |
| --- | --- |
| フレームワーク／標準ライブラリの標準解 | 01 車輪の再発明（整合性↓） |
| プロジェクト内の既存実装・ユーティリティ | 14 隣を見ない再実装（整合性↓） |
| プロジェクトの規約（例外設計・ログ・命名・トランザクション境界） | 13 郷に従わぬ正論（整合性↓） |

## 隣接パターンの切り分け（迷ったら）

| 迷う組み合わせ | 決め手の問い |
| --- | --- |
| 05 場当たり対応 / 06 早合点 | 見せかけの合図を達成と取り違えたか（06）、通すためにわざと合図を緑にしたか（05） |
| 06 早合点 / 11 名ばかりの完了 | 実装が緑・成功を達成と取り違えたか（06）、受け入れ基準が判定可能な粒度になっていないか（11）。弱い緑テストは11の症状として出やすい |
| 02 牛刀をもって鶏を割く / 03 身の丈に合わない設計 | 今の規模に対して技術が過剰か（02）、技術は妥当だが運用する組織がいないか（03） |
| 02 牛刀をもって鶏を割く / 16 取らぬ狸の拡張点 | 今の要件に対して過剰か（02）、将来の口頭の話を根拠にした拡張点か（16） |
| 08 早すぎる抽象化 / 16 取らぬ狸の拡張点 | 抽象化の根拠は「今ある実装1種類」か「将来こうなるかもという口頭の話」か |
| 01 車輪の再発明 / 14 隣を見ない再実装 | 無視したのはフレームワーク・標準ライブラリか、プロジェクト内の既存実装か |
| 07 見切り発車 / 11 名ばかりの完了 | 受け入れ基準そのものが無いか、基準はあるが判定できる粒度になっていないか |
| 03 身の丈に合わない設計 / 04 羹に懲りて膾を吹く | 組織の力を高く見すぎて背伸びしたか、低く見すぎて下方迎合したか |
| 09 制約の後付け / 10 過去への忖度 | 最初から織り込んでいない（順序の話）か、過去に縛られて複雑にした（時相の話）か |

## それでも決まらないとき

どの枝も決め手に欠ける、または複数が同じくらい当てはまるときは、評価軸から逆にたどります。[docs/pattern-catalog.md](pattern-catalog.md) の各パターンの「評価軸の偏り」と、目の前の出力で一番大きく振れている軸を突き合わせます。たとえば制約適合性が振れているなら 09（↓）か 10（↑）、長期の時間効果を高く見積もりすぎているなら 16、変更しやすさを過大に見ているなら 08、と候補を狭められます。

それでも分からなければ、エージェントに `ヒント` と返します。答えは伏せたまま、見る場所→問い→振れている評価軸の順に、段階的に手伝ってくれます（「もっと」でさらに踏み込みます）。一番踏み込んだヒントでも評価軸の名前までで、パターン名は出ません。
