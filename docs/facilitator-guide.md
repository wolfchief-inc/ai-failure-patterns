# 講師向け運用ガイド

## 研修の狙い

[Decision Quality と設計判断失敗パターン](https://scrapbox.io/kawasima/Decision_Quality_%E3%81%A8%E8%A8%AD%E8%A8%88%E5%88%A4%E6%96%AD%E5%A4%B1%E6%95%97%E3%83%91%E3%82%BF%E3%83%BC%E3%83%B3) の16パターンを、座学ではなく「実際にエージェントが踏み抜くコード」を通じて受講者に体感させる。Vibe Coding 時代に、レビュー側の人間が「いまどの失敗パターンに入っているか」を識別できるようになることが目的。

## 受講者前提

- Java と Spring Boot の読み書きができる（フレームワーク慣れの差はあってよい）
- Claude Code（または互換のエージェント環境）が手元で動く
- Scrapbox 原文 or `docs/pattern-catalog.md` を1度通読している

## 研修の流れ（半日 / 1日）

### 半日コース（4時間）

| 時間 | 内容 |
|------|------|
| 30分 | 趣旨説明・カタログ通読・1パターンだけ講師がデモ（演習モード） |
| 60分 | 個人演習: 演習モードで指定3パターンを順に解く（例: 01, 08, 12） |
| 30分 | グループで答え合わせと議論（修正方針の比較） |
| 60分 | 個人演習: テストモードで4回、無作為に解く |
| 30分 | グループ振り返り・ハマったパターンの共有 |
| 30分 | 講師まとめ・現場での `支配軸の取り違え` 検出ポイント |

### 1日コース（7時間）

半日コースに加えて、午後に「グループ対抗ラウンド」と「自分のコードのレビュー実演」を入れる。

## 演習モードでの講師の使い方

- 受講者が `[演習:NN]` でパターンを指定して解く。講師は順序を統制できる
- 序盤は隣接しないパターンを混ぜる（例: 01, 08, 12 のように評価軸の偏りが重ならないもの）
- 後半に隣接パターン（08 早すぎる抽象化 / 16 取らぬ狸の拡張点、または 14 隣を見ない再実装 / 01 車輪の再発明）をぶつけて見分け方を訓練する

## テストモードでの注意

- 受講者プロンプトの内容によって、エージェントが選ぶパターン候補は偏る
- 「Spring Boot で承認ワークフロー」→ sledgehammer / bolt-on-constraints / counting-chickens が出やすい
- 「既存サービスのバグを直す」→ ad-hoc-fix / boundary-violation / rebuild-blind が出やすい
- 「ユーティリティを書く」→ premature-abstraction / wheel-reinvention / castle-in-sand が出やすい
- 講師はプロンプトのお題を変えることで、出現パターンの分布を意図的に動かせる

## 想定難易度

| 難易度 | パターン | 補足 |
|--------|----------|------|
| 易 | 01, 02, 08, 14 | コード上の差分が比較的明確 |
| 中 | 09, 10, 12, 15, 16 | 構造を読まないと気づかない |
| 難 | 03, 04, 06, 07, 11, 13 | 仕様・組織・運用の文脈に踏み込まないと識別できない |

難パターンは座談形式の議論を併せると効く。コード単体では決まらない判断が含まれている。

## 隣接パターンの混線を防ぐ司会の介入

受講者が「この混入は ad-hoc-fix だと思う」と言ってきても、boundary-violation の可能性が見える場合がある。次の質問で切り分ける。

| 切り分け | 質問 |
|----------|------|
| `早すぎる抽象化` vs `取らぬ狸の拡張点` | 抽象化の根拠は「現状の実装1種類」か「将来こうなるかもという口頭発言」か |
| `車輪の再発明` vs `隣を見ない再実装` | 退けられたのは「フレームワーク/標準ライブラリ」か「プロジェクト内の既存実装」か |
| `見切り発車` vs `名ばかりの完了` | 受け入れ基準そのものが無いか、基準はあるが検証可能な粒度になっていないか |
| `場当たり対応` vs `越境実装` | 判断の手前で止まっていないか、責務の置き場所が間違っているか |
| `身の丈に合わない設計` vs `羹に懲りて膾を吹く` | 組織能力を過大評価したか、過小評価したか |
| `制約の後付け` vs `過去への忖度` | 「最初から織り込んでいない」（順序の問題）か「過去に縛られて複雑にした」（時相の問題）か |

## グループ研修でのファシリテーション

- 4〜6人グループ。1人が受講者役で `reveal` 前まで進める。他はレビュー側
- レビュー側は「混入パターン」と「修正方針」を各自先に紙に書いてから議論
- 受講者役が `reveal` を打つ前に、講師が「いま誰が何のパターンに賭けたか」を集計
- reveal 後に正解と賭けの分布を見せる。`隣接パターン` を選んだ人は、なぜそう見えたかを共有してもらう

## エージェントの挙動が崩れる兆候

エージェントの出力が次のいずれかになっていたら、SKILL.md または該当パターンの patterns/examples を見直す。

- 「失敗パターン」「混入」「研修」「Decision Quality」「評価軸」のような語が外向きの応答に漏れている
- 1回の応答で2つ以上のパターンが混入している
- 混入が弱すぎてレビュー側で識別できない（プロにも見えない）
- 混入が強すぎて初心者でも一目でわかる（コードが「明らかに変」）
- 演習モードで指定したパターンと違うパターンが reveal で出る

これらが起きたら、`patterns/<slug>.md` の「混入の指針」「混入してよい局所」「混入してはいけない局所」を厳しくする。

## 受講者によくある誤解

- **「テスト緑になっているのに失敗パターン？」**: テストが緑であることは目的達成と等価ではない（→ 06 早合点）。研修の中心メッセージの1つ
- **「動くんだからいいのでは？」**: 「動く」は最低条件であって設計判断の基準ではない。8軸の評価軸を意識的に取らないと「動くけど壊れる」コードができる
- **「これは AI のせいでは？」**: AI以前から現場にある失敗パターン。AIは件数と頻度を桁違いに増やすだけ。判断を持つ人間がレビュー側にいるかどうかが分かれ目

## 受講後の継続

- 自分のチームの過去のレビューコメントを `支配軸の取り違え` の観点で読み直してもらう
- 自分の PR の説明文に「今回の支配軸は ○○、二次条件は ○○」と書く運用を試してもらう
- `docs/pattern-catalog.md` を週次の輪読資料にして、現場の事例と紐づけてもらう
